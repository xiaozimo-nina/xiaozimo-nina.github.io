7.1 Analysis


Methods of researching an existing system


  • introduce
  • observation
  • interviews
  • questionnaires
  • looking at existing paperwork

introduce


We will consider the first item in the analysis stage -- researching an existing system.
Now we have four common methods used in fact finding, which have been summarised here.

observation


This method involves watching personal using the existing system to find out exactly how it works.


Advantages

  1. The analyst obtains reliable data
  2. It is possible to possible to get an overall view of the system
  3. A relatively inexpensive method

Disadvantages

  1. People are generally uncomfortable being watched and may work in a different way
  2. If workers perform tasks that contravene standard procedures, they may not do this while being watched

interviews


This method involves a one-to-one question-and-answer session between the analyst and the employee/customer. It is a good method if the analyst wants to probe deeply into one specific aspect of the existing system.


Advantages

  1. It gives the opportunity to motivate the interviewee into giving open and honest answers to the analyst's questions
  2. The method allows the analyst to probe for more feedback from the interviewee (it is easier to extend a question)
  3. It is possible to modify questions as the interview proceeds and ask questions specific to the interviewee

Disadvantages

  1. It can be rather a time-consuming exercise
  2. It is relatively expensive (use of analyst's time)
  3. The interviewee can't remain anonymous with this method

questionnaires


This method involves distributing questionnaires to the workforce, clients or system users to find out their views of the existing system and to find out how some of the key tasks are carried out.


Advantages

  1. The questions can be answered quite quickly
  2. It is a relatively inexpensive method
  3. Individuals can remain anonymous if they want
  4. This method allows quick analysis of the data

Disadvantages

  1. The number of returned questionnaires is often low
  2. The questions are rather inflexible since they have to be generic
  3. There is no immediate way to clarify a vague or incomplete answer to a question

looking at existing paperwork


This method allows the analyst to see how the paper files are kept, look at operating instructions and training manuals, check the accounts, ect. It allows the analyst to get some idea of the scale of the problem, memory size requirements, type of input/output devices needed, ect.


Advantages

  1. This method allows information to be obtained which wasn't possible by any of the other methods
  2. Analysts can see for themselves how the paper system operates

Disadvantages

  1. It can be very time-consuming exercise
  2. Because of the analyst's time needed, it is a relatively expensive method to use



Recording and analysing information about the current system


In this case, I have three questions, and let's answer it together!


  1. Why we need to identify the inputs, outputs and processes?
  2. Why we need to identify problems with the current system?
  3. Why we need to identify the user and information requirements for the new system?

Q1

Why we need to identify the inputs, outputs and processes?

  • Every system has inputs and outputs and the systems analyst needs to identify the data input to the present system, and the data output.
  • Any new system that is designed will have to deal with similar inputs and outputs as the present system.

Q2

Why we need to identify problems with the current system?

  • No system is perfect and it is the job of the systems analyst to try and identify where the problems in a system are.
  • If these problems can be fixed, the system will work more smoothly, be more efficient and, in the case of a business, be more profitable.

Q3

Why we need to identify the user and information requirements for the new system?

  • System requirements are the most effective way of meeting the user needs and reducing the cost of implementation.
  • Alao, this could cause a company to save a lot of money and time, and also can cause a company to waste money and time.



System specification


Here we need to know how to identify and justify suitable hardware and software for the new system.



The systems analysts will now need to decide what hardware and software will be required for the new system.

Hardware


  1. How many computers?
  2. What type of network?
  3. How many servers?
  4. Any special input devices?
    (e.g. barcode readers)
  5. Any special output devices?

Software


  1. Is ready-made, off-the-shelf software available?
  2. Is custom-written software required?

Off-the-shelf software


  1. Cheaper
  2. More reliable
    (because most problems will have been found by one of the many users)
  3. Has lots of support and help available
    (because lots of other people are using it)

Custom-written software


  1. Very expensive
  2. Provides exactly what the customer needs
    (a ‘perfect fit’)
  3. Only has one user, so little help is available

Add all these above = Design stage tasks



Paper-based forms need to:

  1. Have a heading to make the purpose of the form clear
  2. Make it clear to the person filling in the form where they must place their answers
  3. Make use of text boxes, which will limit the amount of information collected
  4. Make use of character boxes for data such as surnames, telephone numbers, and so on
    (each box allows one character only)
  5. Make use of printed text boxes to allow for easy input of items such as date of birth
  6. Make use of tick boxes to make choices easier
    (such as sex -- male or female)
  7. Make sure there is sufficient space to write answers
  8. Use clear fonts and clear text colours to ensure the form is east to read

Screen outputs should be designed:

  1. To make sure the size of all the output fields is correct
  2. So that any instructions/descriptions are clear
  3. So that the full screen is utilised (avoiding large areas of 'nothing')
  4. So that colors and fonts (size and type) make the output clear



Validation


Validation is a process where data is checked to see if it satisfies certain criteria when input into a conmuter.
Here are some of the more common checks used when writing computer software.


Validation check

  1. Range check
    Checks whether data is within given/acceptable valus
  2. Look-up check
    Checks whether the data entered exists and is stored in a table of data
  3. Length check
    Checks if the input data contains the required number of characters
  4. Character/type check
    Checks that the input data doesn't contain invalid characters
  5. Format/picture check
    Checks that data is in aspecific format
  6. Presence check
    Checks if data is actually present and hasn't been missed out
  7. Consistency check
    Checls if fields correspond (tie up) with each other
  8. Check digit
    This is an extra digit added to a number which is calculated from the digits


Design and testing strategy/plan


When producting the software for the new system, it is very important to test it thoroughly in order to:

Make suer it meets the agreed client requirements

Remove any bugs/errors from the system

Make sure it produced the required output for data where the correct output is already known

Check that the software doesn't crash under certain conditiond


To do this, it is necessary to produce a testing strategy or plan to ensure all the possible scenarios have been tested so that the criteria have been met.

Implementation


Implementation method
Design of
implementation method
Advantages and disadvantages of the method
Direct With this method, the old system is stoped overnight
and the new system introduced immediately		 This method can be disastrous if the new system fails since the old system is no longer available

The benefits are immediate

Costs are reduced (since only one system used there is no need to pay for teo sets of staff)

Less likelihood of a malfunction since the new system will have been fully tested
Parallel With this method, the old and new systems are run
side by side for a time before the new system
takes over altogether		 It this new system fails, the old system is still available as a backup

It is possible to gradually train staff

It is more expensive than direct since extra staff are needed to run both systems together

It is also more time consuming than direct since data needs to be entered into two systems
Pilot With this method, the new system is introduced into one branch or office of the company and its performance
assessed before being introduced elsewhere in the company		 If the new system fails, only one part is affected; the remainder is unaffacted

It is possible to train staff in one area only, which is much faster and less costly than parallel

The cost are also less than parallel since only one part of the system is being used in the pilot warehouse
Phased With this method, only part of the new system is
introduced and, only when it proves to work
satisfactorily, is the next part introduced, and so on, until the old system is fullt replaced		 If the latest part fails, it is only necessary to go back in the system to the point of failure; hence failure isn't disastrous

More expensive than direct since it is necessary to evaluate each phase before moving to the next stage

Very time consuming since each part needs to be fully evaluated before making any futher changes to the system

7.6 Evaluation


Evaluation


Once a system is up and running it is necessary to do some evaluation and carry out any maintenance if necessary.
The following is a list of some of the things considered when evaluating how well the system has worked;
this can ultimately lead to a redesign of part of the system if there is strong evidence to suggest that changes need to be made:


  1. Compare the final solution with the original task
  2. Identify any limitations of the system
  3. Identify any necessary improvements that need to be made
  4. Evaluate the user's responses to using the new system
  5. Compare test results from the new system with results from the old system
  6. Compare performance of the new system with performance of the old system
  7. Observa users performing set tasks
  8. Measure the time taken to complete tasks
  9. Interview users to gather responses about how well the new system works
  10. Given out questionnaries to gather responses about the ease of use of the new system



Results from the evaluation lead two things happening


Result 1

Update of hardware because

  • Feedback from end-users
  • New hardware comes on the market, necessitating change
  • Changes within the company require new devices to be added or updated

Result 2

Update of software because

  • Feed back from end-users
  • Changes to the company structure or how the company works that may require modifications to the software
  • Changes in legislation that may require modifications to the software
